Flock can run in the browser as a DuckDB-WASM loadable extension, using the same SQL API as the native extension. This enables client-side analytics and demos without server infrastructure.

## Native vs browser

| Environment | Install |
|-------------|---------|
| **CLI / Python / etc.** | [Installation](/installation) |
| **DuckDB-WASM** | Same commands as [Installation](/installation) — extensions are fetched as `.wasm` modules on `LOAD` |

See [Getting Started](/getting-started/overview) for provider configuration after install.

## Using Flock in DuckDB-WASM

In a DuckDB-WASM environment (for example [shell.duckdb.org](https://shell.duckdb.org) or an app using [@duckdb/duckdb-wasm](https://duckdb.org/docs/stable/clients/wasm/overview)), run the install commands from [Installation](/installation).

In WASM, `INSTALL` records extension metadata; the `.wasm` binary is fetched when you `LOAD`. See the [DuckDB-WASM extensions guide](https://duckdb.org/docs/stable/clients/wasm/extensions) for details.

After loading, use Flock functions (`llm_complete`, `llm_filter`, etc.) the same way as in the CLI — subject to the limitations below.

## Building WASM from source

Flock CI builds WASM artifacts via [DuckDB extension-ci-tools](https://github.com/duckdb/extension-ci-tools) (see `.github/workflows/MainDistributionPipeline.yml`). To build locally:

1. Clone with submodules and follow the [Developer Guide](/developer-guide).
2. Use the extension distribution toolchain with `WASM_EXTENSIONS=1` and an Emscripten target, following the patterns in `extension-ci-tools/.github/workflows/_extension_distribution.yml`.

Published WASM binaries are distributed through DuckDB's extension repository when released to the community catalog.

## Limitations

<Warning>
Browser usage has constraints that do not apply to native DuckDB:
</Warning>

- **API keys**: Provider credentials must be available to the browser session. Avoid embedding production secrets in client-side code.
- **Network**: LLM calls still go to external provider APIs; only DuckDB computation runs locally.
- **CORS**: Browser fetches to provider endpoints may be blocked unless your app proxies requests.
- **Secrets persistence**: DuckDB-WASM has no durable cross-session storage for `CREATE SECRET`; configure secrets per session.
- **Feature parity**: Multimodal and provider support matches the WASM build you load; verify your target platform in CI release artifacts.

For production browser apps, consider a backend that holds API keys and exposes a controlled SQL or API surface.
